\section{Metodologías}
Para llevar a cabo el proyecto se han seguido diversas metodologías de desarrollo ágil. Estas metodologías definen la forma en la que se planifica, organiza y gestiona el desarrollo de un proyecto, permitiendo estructurar las tareas, coordinar el progreso y mejorar la calidad del resultado.

\subsection{\eng{Scrum}}
La metodología \eng{Scrum} es un marco de trabajo ágil orientado a la gestión y desarrollo de proyectos de manera iterativa e incremental. Se organiza en ciclos cortos denominados \eng{sprints}, al final de los cuales se entrega un incremento funcional del producto (ver imagen~\ref{fig:Memoria/Metodologia_SCRUM} para ver el proceso \eng{Scrum} completo). Estos \eng{sprints} favorecen la mejora continua y la adaptación al cambio, además, aseguran una entrega temprana de valor. A diferencia de metodologías tradicionales, \eng{Scrum} aporta mayor flexibilidad y retroalimentación constante~\cite{scrum:guide}. 

\imagenflotante{Memoria/Metodologia_SCRUM}{Metodología \eng{Scrum}.}{1}

\subsection{\eng{Kanban}}
Para la gestión del flujo de trabajo y organización de tareas se ha adoptado la metodología \eng{Kanban}~\cite{herramienta:kanban}. La cual consiste en un enfoque ágil orientado a la visualización del trabajo, la optimización del flujo y la mejora continua. Se basa en la idea de que el trabajo debe gestionarse como un flujo continuo en lugar de dividirse en iteraciones cerradas. Se centra en representar el estado de las tareas de forma visual, lo que facilita la planificación, el seguimiento y la detección de cuellos de botella. Este enfoque es útil en proyectos de desarrollo \eng{software} en los que los requisitos puedan evolucionar, haciendo necesario tener una planificación flexible.

Finalmente, al tratarse de la plataforma más utilizada a nivel mundial, cuenta con abundante documentación, lo que la convierte en una alternativa sólida y con amplio soporte~\cite{gitHub}.

\section{Gestión de dependencias y entornos \eng{Python}}
El proyecto requiere entornos aislados y reproducibles que faciliten el trabajo local, la integración continua y la generación de entregables. Se ha valorado utilizar \myhref{http://python-poetry.org/docs/}{\eng{Poetry}} y \myhref{https://docs.astral.sh/uv/}{\eng{uv}}. \eng{Poetry} es una herramienta madura de gestión de dependencias y empaquetado que centraliza la configuración en \file{pyproject.toml}. \subsection{Base de datos vectorial}\label{basedatos_vectorial}

Presenta equilibrio entre el rendimiento, la latencia y el tiempo de indexado~\cite{llmeng:handbook}. Es útil cuando la aplicación ya se apoya en este gestor, ya que simplifica la infraestructura~\cite{Qdrant:pgvector}. \section{Desarrollo \eng{web}}
El desarrollo de la interfaz \eng{web} para el proyecto se ha realizado con \myhref{https://flask.palletsprojects.com/en/stable/}{\eng{Flask}}~\cite{herramienta:flask}. \eng{Flask} es un \eng{framework} WSGI (\eng{Web Server Gateway Interface}~\cite{herramienta:wsgi} es un estándar \eng{Python} para la comunicación entre los servidores \eng{web} y el \eng{framework web}) que proporciona los componentes esenciales para construir aplicaciones \eng{web} usando \eng{Python}. Fue diseñado para la creación de aplicaciones \eng{web} ligeras y modulares. 

Se ha optado por utilizar \eng{Flask} para el desarrollo \eng{web} en lugar de otros \eng{frameworks}, como \myhref{https://www.djangoproject.com/}{\eng{Django}} o \myhref{https://fastapi.tiangolo.com/}{\eng{FastAPI}} porque incorpora una serie de componentes que permiten gestionar las aplicaciones \eng{web} de forma sencilla:
\begin{itemize}
\tightlist
	\item Enrutamiento: permite asociar rutas URL a funciones \eng{Python} utilizando decoradores. Lo cual permite organizar el código de manera modular. En el caso del proyecto simplifica la conexión entre la interfaz \eng{web} y el \eng{pipeline} de generación \eng{RAG}.
	\item Gestión de peticiones y respuestas: proporciona objetos que permiten acceder de manera sencilla a los datos enviados por el usuario, utilizando formularios, \file{JSON} o simplemente parámetros.
	\item Motor de plantillas \file{Jinja2}: permite generar \file{HTML} a partir de datos procesados en el servidor. Lo cual facilita la visualización estructurada de las distintas partes del sistema.
	\item Soporte WSGI: como se ha comentado anteriormente, \eng{Flask} se basa en este estándar para la comunicación entre aplicaciones \eng{web} y servidores \eng{Python}. La utilización de este estándar garantiza la compatibilidad con múltiples servidores de producción y facilita su despliegue mediante \eng{Docker}.
	\item Extensiones: a pesar de que es minimalista, permite incorporar extensiones para bases de datos, autenticación o validación de formularios.
\end{itemize}

\subsection{Docker}\label{docker}
\myhref{https://www.docker.com/}{\eng{Docker}}~\cite{herramienta:docker} es una plataforma de \eng{software} de código abierto basada en contenedores. Permite empaquetar una aplicación con sus dependencias para distribuirla y ejecutarla, asegurando que funcione igual en cualquier entorno. Esto se debe a que aísla la aplicación y sus dependencias del sistema operativo dentro de unidades ligeras y portables (los contenedores). Los conceptos clave del \eng{Docker} son:
\begin{itemize}
\tightlist
	\item Imagen: es una plantilla que contiene el sistema base y las dependencias necesarias.
	\item Contenedor: es una instancia en ejecución de la imagen.
	\item \file{Dockerfile}: es un archivo en el que se especifican los pasos que se deben seguir para construir la imagen.
	\item \file{Docker Compose}: es una herramienta que se define en un archivo \file{yml} en el cual se definen todos los servicios que debe contener la imagen. Estos servicios se dividen en volúmenes. 
\end{itemize} 

Se decidió implementar un contenedor \eng{Docker} ya que mejora la reproductividad, la portabilidad y evita conflictos entre las dependencias en el sistema aportando mayor aislamiento. Además, permite integrar fácilmente diversas herramientas dentro de los volúmenes lo que mejora la escalabilidad. En sistemas que combinan múltiples componentes, como ocurre en este proyecto con LLM, base de datos vectorial y relacional e interfaz web, el uso de contenedores simplifica considerablemente la gestión de la infraestructura.

\subsection{\eng{Nginx}}\label{nginx}
\myhref{https://nginx.org/}{\eng{Nginx}} es un servidor \eng{web} que puede actuar como \eng{proxy} inverso~\cite{herramienta:proxyinverso}. Un \eng{proxy} inverso es un servidor que, a diferencia de un \eng{proxy} normal, se sitúa delante de los servidores \eng{web} para interpretar las solicitudes de los clientes. Se encarga de reenviar las solicitudes de los clientes a los servicios \eng{web}. En el caso del proyecto \eng{Nginx} se ha utilizado como intermediario entre el cliente y la aplicación \eng{Flask} para recibir las peticiones externas (HTTP/ HTTPS). También, se encarga de gestionar el certificado SSL y de redirigir el tráfico hacia \eng{Gunicorn}. 

La ventaja de utilizar \eng{Nginx} es que está optimizado para manejar múltiples conexiones concurrentes de forma eficiente. Además, desacopla el servidor \eng{web} del servidor de aplicaciones, aumentando la seguridad y el rendimiento del sistema.
\subsection{\eng{Gunicorn}}\label{gunicorn}
\myhref{https://gunicorn.org/}{\eng{Gunicorn}} es un servidor WSGI para aplicaciones \eng{Python} fácil de instalar (no requiere dependencias adicionales), pero que no es compatible con \eng{Windows}. Su principal función es ejecutar la aplicación \eng{Flask} en un entorno preparado para producción. Ya que, por defecto, el \eng{framework} \eng{Flask} incluye un servidor de desarrollo que no es adecuado para entornos reales. Por lo cual \eng{Gunicorn} se encarga de sustituir este servidor por uno optimizado para producción que si que va a permitir gestionar múltiples procesos de manera simultánea y que va a mejorar la estabilidad. 

\subsection{DNS}\label{dns}
El DNS (\eng{Domain Name System}) permite asociar un nombre de dominio a la dirección IP del servidor donde se encuentra desplegada la aplicación. Asignarle un dominio a una aplicación sirve para facilitar el acceso y otorgarle mayor profesionalidad. Además, de mejorar la accesibilidad, permite la migración del servidor sin necesidad de cambiar la dirección que utiliza el usuario.

En este proyecto el dominio público se ha adquirido a través del proveedor \myhref{https://www.hostalia.com/}{\eng{Hostalia}}. Una vez registrado el dominio, fue necesario configurar sus registros DNS para asociarlo con la dirección IP del servidor donde se encuentra desplegada la aplicación. Para vincular el dominio al servidor se configuró un registro tipo A dentro del panel de gestión DNS de \eng{Hostalia}. Este registro permite que las peticiones dirigidas al dominio se redirijan a la dirección IP correspondiente del servidor.

Sin embargo, la aplicación se ejecuta en una IP privada de la Universidad de Burgos. Las direcciones IP privadas no son accesibles directamente desde Internet~\cite{ herramienta:ip_privadaVSpublica}, ya que están destinadas a redes internas. Para conseguir que la aplicación sea accesible desde Internet hay que configurar el \eng{router} de tal forma que el dominio apunte a la IP pública del dicho \eng{router}. Mediante técnicas NAT (\eng{Network Address Translation}~\cite{herramienta:nat}) se redirige el tráfico hacia la IP privada del servidor. 

\subsection{Certificado SSL/TLS}\label{https}
Para garantizar la seguridad de las comunicaciones se ha configurado un certificado SSL/TLS que permite el uso del protocolo HTTPS (\eng{HyperText Transfer Protocol Secure})~\cite{herramienta:HTTPvsHTTPS}. HTTPS es la versión segura de HTTP y se basa en el protocolo TLS (\eng{Transport Layer Security}), cuyo objetivo es cifrar la comunicación entre el cliente y el servidor. Este cifrado impide que terceros puedan interceptar, modificar o suplantar la información transmitida, protegiendo así la confidencialidad e integridad de los datos.

En este proyecto, el certificado digital se ha emitido utilizando \myhref{https://certbot.eff.org/}{\eng{Certbot}} que es una herramienta oficial promovida por la \myhref{https://www.eff.org/}{\eng{Electronic Frontier Foundation}} (EFF). \eng{Certbot} automatiza la obtención e instalación de certificados gratuitos proporcionados por \myhref{https://letsencrypt.org/}{\eng{Let’s Encrypt}}. La cual es una autoridad certificadora reconocida internacionalmente. El proceso de emisión del certificado se basa en un mecanismo de validación del dominio, mediante el cual se demuestra que el servidor tiene control sobre el dominio solicitado. Una vez validado, se genera un certificado digital que incluye la clave pública del servidor, la identidad del dominio y la firma digital de la autoridad certificadora. Concretamente se ha utilizado el método de validación DNS-01 que consiste en demostrar que se tiene control sobre el dominio añadiendo un registro específico tipo \file{TXT} en la configuración DNS. Este registro contiene un \eng{token} generado por la autoridad certificadora.
