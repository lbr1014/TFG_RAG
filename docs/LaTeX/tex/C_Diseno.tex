\section{Diseño de datos}

Para llevar a cabo la aplicación se han definido las siguientes entidades:
\begin{itemize}
\tightlist
	\item Usuario: esta entidad representa a la persona que usa la aplicación. Distingue entre usuario normal y administrador, al cual le permite realizar la gestión de documentos y usuarios. Esta entidad maneja un identificador (id) que es la clave primaria, un nombre, un email que debe ser único, una contraseña que se almacena encriptada, el último inicio de sesión y el rol.
	\item Consulta: esta entidad guarda las preguntas que realizan los usuarios, así como las respuestas que da el \eng{LLM} alimentado por el \textit{RAG}. Es la base para el historial de consultas donde se van a listar y borrar. Esta entidad maneja un identificador (id) que es la clave primaria, el identificador del usuario que realiza la consulta, la pregunta del usuario, la respuesta, los fragmentos utilizados, el tiempo de respuesta, y la fecha en la que se preguntó al modelo.
	\item Documento: esta entidad representa un PDF cargado en la aplicación. Maneja un identificador (id) como clave primaria, el nombre del fichero, el tamaño, la fecha de modificación, el número de \eng{chunks} en la base de datos vectorial, el \textit{hash} del documento y el estado del documento (cargado, procesado, indexado, fallido). 
	\item \eng{Chunk}: esta entidad representa cada trozo de texto en el que se dividen los documentos para poder generar los \textit{embeddings}. Maneja el texto que forma al propio \textit{chunk}, el \textit{embedding} y los metadatos del \textit{chunk} (nombre del archivo, título y posición dentro del documento).
	\item \eng{Embedding}: esta entidad se corresponde con la representación vectorial utilizada para la base de datos vectorial. Va a permitir recuperar \textit{chunks} por similitud. La entidad maneja un vector asociado al \textit{chunk}.
	\item \eng{ConsultaChunk}: esta entidad modela la relación entre las consultas y los \textit{chunks} permitiendo que una consulta tenga varios \textit{chunk} y que un \textit{chunk} pueda servir para varias consultas. La entidad maneja el identificador de la consulta, el identificador del \textit{chunk}, la similitud con la consulta del usuario y el \textit{ranking}. El \eng{ranking} hace referencia a la relevancia que tiene un \eng{chunk} entre los recuperados para aumentar el \eng{prompt} del \eng{LLM}. La posición del \eng{chunk} en el \eng{ranking} se determina ordenandolos descendentemente según su similitud.
\end{itemize}

Las relaciones entre las entidades son:
\begin{itemize}
\tightlist
	\item Un usuario puede hacer cero o varias consultas.
	\item Cada consulta pertenece a un usuario.
	\item Un documento tiene cero (si está cargado pero no indexado) o varios \eng{chunks}.
	\item Cada \eng{chunk} pertenece a un documento.
	\item Un \eng{chunk} tiene un \textit{embedding}.
	\item Cada \eng{embedding} pertenece a un \textit{chunk}.
	\item Una consulta usa uno o varios \eng{chunk}.
	\item Un \eng{chunk} puede ser usado por cero o varias consultas.
\end{itemize}

\section{Diseño arquitectónico}
El objetivo del diseño arquitectónico es garantizar la modularidad, la escalabilidad, la mantenibilidad, la portabilidad y la reproducibilidad del sistema desarrollado. Es decir, se busca tener una separación clara de responsabilidades entre la interfaz, la lógica, la persistencia y los servicios de Inteligencia Artificial con una organización del código en componentes cohesivos (\eng{Blueprints}, servicios y modelos), pudiendo replicarse los servicios de manera sencilla. Además, sustituir los componentes debe tener un impacto mínimo en el resto del código. 

Este sistema se basa en una arquitectura cliente–servidor multicapa, combinada con una arquitectura \eng{RAG} (\eng{Retrieval-Augmented Generation}), en la capa de aplicación, estructurada en dos \eng{pipelines} diferenciados, el de indexación y el de generación. El despliegue se lleva a cabo utilizando contenedores \eng{Docker} para almacenar el código junto con sus dependencias.

\subsection{Arquitectura general del sistema y despliegue}
El sistema sigue una arquitectura distribuida basada en servicios (\eng{service-oriented}~\cite{diseño:SOA}), donde cada componente tiene una responsabilidad claramente definida. La aplicación se compone de los siguientes elementos principales:
\begin{itemize}
\tightlist
	\item Cliente \eng{web}: se corresponde con el navegador del usuario.
	\item Servidor de \eng{proxy} inverso \eng{Nginx}: el cual se encarga del protocolo de seguridad a través del certificado SSL/TLS, las redirecciones HTTP a HTTPS y el encaminamiento hacia la aplicación .
	\item Capa de aplicación: aplicación \eng{Flask} desplegada tras un servidor WSGI \eng{Gunicorn}.
	\item Persistencia relacional: se incluye \eng{PostgreSQL} como base de datos relacional para asegurar la persistencia de los datos de las transacciones, como son los usuarios, las consultas, los documentos o los estados.
	\item Persistencia vectorial: se utiliza \eng{Qdrant} como base de datos vectorial para realizar la búsqueda semántica sobre los \eng{embeddings}.
	\item Servicio de modelos \eng{LLM}:  se utiliza \eng{Ollama} para servir el modelo \eng{LLM} de manera local exponiendo una API HTTP de generación.
	\item Contenedor de pruebas: se utiliza este contenedor para la ejecución de las pruebas del sistema, así como, de la cobertura de dichas pruebas.
\end{itemize}

El sistema se despliega mediante contenedores \eng{Docker}. Los servicios se definen en el archivo \file{Docker Compose}. Se definen seis contenedores independientes que se corresponden con el entorno para la base de datos relacional \eng{PostgresSQL} llamado \file{db}, el de la base de datos vectorial \eng{Qdrant} llamado \file{qdrant}, el de el servicio \eng{LLM} llamado \file{Ollama}, el de la aplicación \eng{Flask} levantada usando \eng{Gunicorn} llamado \file{web}, el del \eng{proxy} inverso \file{nginx} y el contenedor de ejecución de pruebas llamado \file{test}.

Gracias a esta separación se garantiza la seguridad ya que no se expone directamente la aplicación \eng{Flask}. Este despligue también, aumenta la escalabilidad, la reproducibilidad y la portabilidad al tener la posibilidad de replicar los servicios en un entorno aislado que no depende del sistema anfitrión.

La comunicación entre componentes (ver imagen ~\ref{fig:Arquitectura/ComunicacionArquitectura}) se realiza principalmente dentro de al red interna de \file{Docker-compose}, donde los servicios se resuelven entre sí mediante el nombre de servicio,es decir, el DNS interno, y se conectan a través de los puertos expuestos internamente. El acceso desde el exterior se gestiona usando \eng{Nginx}, que es el único componenete que publica puertos hacia internet (el puerto 80 para el protocolo HTTP y el 443 para el HTTPS). \eng{Nginx} actúa como \eng {proxy} inverso y punto de terminación TLS. Recibe las peticiones que hacen los usuarios a la \eng{web} \file{pythia.es} y las reenvía por HTTP interno al servicio \eng{web} en \file{web:5000} que se corresponde con la aplicación \eng{Flask} servida con \eng{Gunicorn}. Desde la aplicación, las comunicaciones con los servicios de datos se realizan dentro del entorno \eng{Docker}. La conexión con la base de datos relacional \eng{PostgreSQL} se gestiona con una conexión TCP a \file{db:5432}. La conexión con la base de datos vectorial \eng{Qdrant}, para la recuperación semántica, se realiza mediante conexión HTTP por el puerto \file{qdrant:6333}. Para generar las respuestas se invoca el servicio \eng{Ollama} por HTTP en \file{ollama:11434}. \eng{Ollama} puede ser accesible desde fuera del \eng{host} gracias al mapeo de puertos \file{11435:11434}. Por último, el contenedor \eng{ollama-init} se comunica con \eng{Ollama} de forma interna durante el arranque para descargar el modelo y dejarlo disponible antes de su uso. En el diagrama~\ref{fig:Arquitectura/Peticiones} se puede observar como atiende el sistema las peticiones de los usuarios.

\subsection{Arquitectura cliente-servidor multicapa}
Desde el punto de vista lógico el sistema se estructura siguiendo una arquitectura cliente-servidor multicapa, organizada en tres niveles claramente diferenciados. Estos niveles se corresponden con la capa de presentación, la capa de aplicación y la capa de persistencia y servicios externos (ver arquitectura en la imagen~\ref{fig:Arquitectura/ArquitecturaLogica}.

\imagendiagram{Arquitectura/ArquitecturaLogica}{Arquitectura cliente-servidor multicapa.}{1.1}

\begin{itemize}
\tightlist
	\item \textbf{Capa de presentación}: es la interfaz de usuario (IU) del sistema y es accesible a través del navegador web. Se sigue un enfoque SSR (\eng{Server-Side Rendering}), en el que las páginas HTML se generan en el servidor mediante plantillas (usando las plantillas \file{Jinja2} en \eng{Flask}) y se envían al navegador (cliente) ya renderizadas. Esto reduce la complejidad en el cliente, simplifica el control de acceso y la validación en el servidor. Mejora la trazabilidad y el registro de las interacciones y simplifica la integración del \eng{pipeline RAG}.
	\item \textbf{Capa de aplicación}: se implementa en el contenedor \eng{web}, que es donde se ejecuta la aplicaicón \eng{Flask} servida pro \eng{Gunicorn}. Esta capa es responsable de coordinar la lógica funcional del sistema y actúa como núcleo del procesamiento. Esta compuesta por los controladores (rutas) organizados en \eng{Blueprints} encargados de gestionar las peticiones HTTP, la lógica de negocio (gestión de usuarios, documentos, consultas, etc) y el \eng{pipeline RAG}.
	\item \textbf{Capa de persistencia} y \textbf{servicios externos}: en esta capa se agrupan los sistemas encargados del almacenamiento y procesamiento de los datos. Se incluyen las bases de datos, tanto la relacional \eng{PostgreSQL} como la vectorial \eng{Qdrant}. También, se incluye \eng{Ollama} como proveedor del modelo \eng{LLM}. Todos estos servicios se ejecutan como contenedores independientes dentro de la red interna de \eng{Docker}, garantizando aislamiento, modularidad y facilidad de despliegue.
\end{itemize}

\subsection{Arquitectura de la aplicación \eng{web}}
La aplicación \eng{web} sigue una arquitectura MVC (\eng{Model-View-Controller}). Esta arquitecturaes típica de las aplicaciones \eng{Flask} \eng{Server-Side-Rendered} (SSR)~\cite{diseño:SSR}. La técnica SSR consiste en generar el código de las páginas en el lado del servidor y entregar al cliente el código \file{HTML} completamente montado con toda la información que se debe mostrar. 

El modelo vista intención (MVC)~\cite{diseño:MVC} es un patrón arquitectónico de software que separa el código en tres capas independientes. La capa modelo donde se incluye la lógica y los datos , la capa vista donde se implementa la interfaz de usuario (\eng{frontend}) y la capa controlador que actúa de intermediario entre las acciones del usuario en las vistas y la lógica de la aplicación (rutas). En esta aplicación se implementa adaptado al ecosistema \eng{Flask} (ver diagrama~\ref{fig:Arquitectura/ModeloVistaControlador}).

\imagendiagram{Arquitectura/ModeloVistaControlador}{Diagrama de la arquitectura Modelo-Vista-Controlador (MVC) del sistema.}{1.1}

\begin{itemize}
\tightlist
	\item La \textbf{vista}: está compuesta por las platillas \file{HTML} renderizadas mediante el motor de plantillas \file{Jinja2}, responsables de presentar la interfaz y capturar las interacciones del usuario. Además, incorpora los recursos estáticos (\file{css} y \file{JavaScript}). Aunque, la aplicación incorpora \eng{JavaScript} para mejorar la experiencia de usuario, la generación principal del contenido es \eng{server-side}, por lo que la arquitectura sigue siendo fundamentalmente SSR.
	\item El \textbf{controlador}: se implementa mediante rutas \eng{Flask} organizadas en \eng{Blueprints}. El uso de \eng{Blueprints} permite mantener alta cohesión dentro de cada módulo y bajo acoplamiento entre funcionalidades. Se encarga de gestionar las peticiones HTTP, realizar validaciones básicas (como las de los formularios), así como controlar de acceso por rol. También, tiene la responsabilidad de invocar los servicios donde se encuentra la lógica, coordinar el \eng{pipeline} \eng{RAG} y generar las respuestas HTTP (renderizando la vista adecuada) . 
	\item El \textbf{modelo}: esta compuesto por las entidades de datos persistentes (Usuario, Consulta, Documento, \eng{Chunk}, \eng{Embedding} y la relación \eng{ConsultaChunk}) y por la lógica de negocio asociada (gestión de usuarios, administración documental, \eng{pipelines} de consulta e indexación \eng{RAG}, control de estados de procesos largos, etc). Se usa una base de datos relacional (\eng{PostgresSQL}) para asegurar la persistencia de los datos de la aplicación y una base de datos vectorial (\eng{Qdrant}) para la recuperación semántica. Esta capa permite desacoplar la lógica de negocio de la base de datos subyacente.
\end{itemize} 

Con estas capas se mantiene el desacoplamiento de la aplicación \eng{web}, haciendo que los controladores organicen el flujo para que la interfaz de usuario no implemente reglas de negocio y sea el modelo el que encapsule los datos y los procesos del domino. 

\subsection{Arquitectura del sistema \eng{RAG}}
El sistema \eng{RAG} se integra dentro de la capa de aplicación. Es el encargado de integrar en el sistema la generación de respuestas. u diseño se basa en la combinación de recuperación semántica sobre una base de datos vectorial y generación de texto mediante un modelo de lenguaje (\eng{LLM}). A nivel de arquitectura se divide en dos \eng{pipelines}, el de indexación y el de generación (ver diagrama~\ref{fig:Arquitectura/ArquitecturaRAG}).

\imagendiagram{Arquitectura/ArquitecturaRAG}{Diagrama de la arquitectura del modelo \eng{RAG}.}{1.1}

El \textbf{\eng{pipeline} de indexación} se ejecuta cuando se cargan los documentos en la aplicación, ya bien sea de manera manual o mediante \eng{web scraping}. Por cada documento se va a llevar a cabo una conversión a \file{Markdown} para que todos tengan una estructura similar que el \eng{LLM} entienda fácilmente. De estos documentos se extrae, limpia y normaliza su texto, posteriormente se segmenta en \eng{chunks} con solapamiento y se generar sus \eng{embeddings}. Los cuales se van a almacenar en la base de datos vectorial (\eng{Qdrant}). Este proceso se ejecuta de forma asíncrona para evitar bloquear la interfaz \eng{web}. Cuando acaba la indexación de los documentos se envía un correo electrónico al usuario parta notificárselo.

El \textbf{\eng{pipeline} de generación} se ejecuta cuando el usuario hace una consulta. Su objetivo es dar una respuesta contextualizada utilizando los \eng{chunks} ya indexados en el sistema. Para ello va a generar los \eng{embeddings} de la consulta y a recuperar los \eng{chunks} más relevantes para generar la respuesta de \eng{Qdrant}. Los \eng{chunks} más relevantes se calculan por la similitud de sus \eng{embeddings} conlos de la consulta. Los usa para construir el \eng{prompt} aumentado, el cual envía al modelo \eng{LLM} para que responda a la consulta. Finalmente, muestra su respuesta al usuario y registra la consulta en la base de datos relacional. Este \eng{pipeline} combina recuperación semántica y generación de texto, asegurando trazabilidad al devolver los metadatos de los documentos fuente.
