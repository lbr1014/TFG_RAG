\section{Manual del programador}
\label{prog:manual}
Este apartado tiene como objetivo proporcionar a futuros desarrolladores que necesiten desplegar, modificar o ampliar la aplicación.

El proyecto implementa una aplicación basada en una arquitectura RAG (\eng{Retrieval Augmented Generation}. Orientada a la gestión y consulta inteligente de documentación, utilizando tecnologías de procesamiento de lenguaje natural, recuperación semántica y generación de respuestas mediante modelos de inteligencia artificial.

El sistema ha sido desarrollado principalmente en \file{Python} utilizando el \eng{framework} \file{Flask} para la capa \eng{web} y diferentes librerías relacionadas con \eng{embeddings}, recuperación documental y modelos de lenguaje. Además, se emplea \file{Docker} para facilitar la ejecución del entorno y garantizar la reproducibilidad del sistema.

El desarrollo del proyecto se ha realizado utilizando las siguientes herramientas y tecnologías:
\begin{itemize}
\tightlist
	\item Lenguaje de programación: \file{Python 3}.
	\item \eng{Framework} \eng{backend}: \file{Flask}.
	\item Base de datos relacional: \file{SQLite/MySQL} según entorno.
	\item Sistema de migraciones: \file{Flask-Migrate}.
	\item Sistema de plantillas: \file{Jinja2}.
	\item Gestión de dependencias: \file{pip}.
	\item Contenedores: \file{Docker} y \file{Docker Compose}.
	\item Control de versiones: \file{Git} y \file{GitHub}.
	\item Entorno de desarrollo: \file{Visual Studio Code}.
	\item Librerías de IA y recuperación semántica: \file{LangChain}, modelos de \eng{embeddings} y \file{Qdrant}.
\end{itemize}

La estructura general del proyecto se encuentra organizada siguiendo una arquitectura modular que separa la lógica de negocio, las rutas, los formularios y las funcionalidades relacionadas con el sistema \eng{RAG}. Concretamente la aplicación sigue una arquitectura cliente-servidor multicapa. La capa de presentación está formada por las plantillas \file{HTML} renderizadas mediante \file{Flask} y \file{Jinja2}, mientras que la lógica de negocio se implementa en los controladores y servicios internos. El componente principal del proyecto es el sistema \eng{RAG}. Este sistema combina recuperación semántica de información y generación de respuestas utilizando modelos de lenguaje.

El núcleo del sistema se basa en un flujo \eng{RAG} compuesto por las siguientes etapas:
\begin{enumerate}
\tightlist
	\item Carga y procesamiento de documentos.
	\item Conversión de documentos a formato \file{Markdown}.
	\item Fragmentación del contenido textual.
	\item Generación de \eng{embeddings} semánticos.
	\item Indexación de fragmentos en la base vectorial.
	\item Recuperación de contexto relevante ante una consulta.
	\item Generación de respuesta mediante un modelo de lenguaje.
\end{enumerate}

Por otra parte, la aplicación implementa un sistema de \file{Blueprints} de \file{Flask} para dividir las distintas áreas funcionales del sistema.

\begin{enumerate}
\tightlist
	 \item Módulo de autenticación (\file{auth}): es el encargado de registrar usuarios, el inicio y cierre de sesión, la recuperación de contraseña, la gestión de sesiones y la validación de credenciales.
	 \item Módulo principal (\file{main}): se encarga de la página principal, así como de la navegación general. Gestiona el historial de las consultas y permite la visualización de la información del usuario.
	 \item Módulo RAG (\file{rag}): contiene la lógica para subir documentos, convertirlos a \file{Markdown}, procesar semánticamente, recuperar el contexto para la consulta del usuario y generar las respuestas llamando a un \eng{LLM}.
	 \item Módulo de administración (\file{admin}): permite gestionar a los usuarios, asignarles permisos y administrar el sistema (carga de documentos, actualización de la base de datos vectorial, etc).
\end{enumerate}

En la figura~\ref{fig:Diagramas/manual_programador_organizacion_codigo.pdf} se muestra la organización general de los principales componentes \eng{software} de la aplicación y las relaciones existentes entre ellos. El hecho de que se implemente una arquitectura modular permite añadir nuevas funcionalidades de manera sencilla. Para añadir una nueva funcionalidad se recomienda crear un nuevo \file{Blueprint} encargado de gestionar las rutas y peticiones \file{HTTP} asociadas. La lógica de negocio debe implementarse en la capa de servicios, manteniendo desacoplados los controladores de la funcionalidad interna. Si la nueva funcionalidad requiere persistencia, se pueden incorporar nuevos modelos \file{SQLAlchemy} y generar las migraciones correspondientes mediante \file{Flask-Migrate}. Del mismo modo, es posible integrar nuevos servicios externos, modelos de lenguaje, sistemas de procesamiento documental o mecanismos de recuperación de información reutilizando la estructura existente.

El flujo de datos del procesamiento documental tras la carga de un documento por parte de un usuario en el sistema es:

\begin{enumerate}
\tightlist
	\item Validación del archivo.	
	\item Conversión del documento a texto \file{Markdown}.
	\item Limpieza y normalización del contenido.
	\item División en fragmentos de tamaño controlado.
	\item Generación de \eng{embeddings} para cada fragmento.
	\item Almacenamiento de \eng{embeddings} en la base vectorial.
\end{enumerate}

El sistema mantiene la trazabilidad entre los fragmentos indexados y el documento original.

El flujo de datos de la recuperación semántica ante una consulta del usuario (ver imagen~\ref{fig:Diagramas/diagrama_flujo}):

\begin{itemize}
\tightlist
	\item Se genera un \eng{embedding} de la pregunta.
	\item Se realiza una búsqueda vectorial.
	\item Se recuperan los fragmentos más relevantes.
	\item El contexto recuperado se envía al modelo de lenguaje.
	\item El modelo genera una respuesta contextualizada.
\end{itemize}

Este enfoque permite reducir alucinaciones y mejorar la precisión de las respuestas generadas.

El núcleo de la aplicación se ha desarrollado utilizando \file{Flask} y diversas extensiones que facilitan la implementación de funcionalidades habituales en aplicaciones \eng{web}.

\begin{itemize}
\tightlist
	\item \file{Flask}: \eng{framework web} utilizado para gestionar rutas, peticiones \file{HTTP} y el renderizado de contenido dinámico.
	\item \file{Flask-Login}: gestión de autenticación, sesiones de usuario y control de acceso.
	\item \file{Flask-WTF} y \file{WTForms}: definición, validación y procesamiento de formularios \eng{web}.

\section{Pruebas del sistema}
\label{prog:prueb}

Con le objetivo de comprobar el correcto funcionamiento de la aplicación \eng{web} se ha desarrollado un conjunto de pruebas automáticas. Concretamente se han realizado pruebas unitarias, de integración y de interfaz que permiten validar tanto el funcionamiento individual de los componentes del sistema como la integración entre ellos.

\subsection{Pruebas unitarias}

Las pruebas unitarias tienen como objetivo verificar de forma aislada el comportamiento de funciones, métodos y componentes individuales de la aplicación. Para su implementación se ha utilizado el \eng{framework} \file{unittest}. Estas pruebas se centran en validar la lógica de negocio de la aplicación, comprobando aspectos como la autenticación de usuarios, la gestión documental, la validación de formularios, la generación de respuestas, la manipulación de datos y las funciones auxiliares utilizadas por el sistema.

Concretamente, se han desarrollado pruebas unitarias para los distintos módulos funcionales de la aplicación. Entre ellas destacan las pruebas asociadas a los modelos de datos, los formularios de autenticación, los servicios de validación de correos electrónicos, los mecanismos de internacionalización, la gestión documental, la conversión de documentos a \file{Markdown}, el sistema de \eng{Web Scraping} y los servicios encargados de la recuperación semántica y generación de respuestas del sistema \eng{RAG}.

También, se han implementado pruebas específicas para los módulos de evaluación automática basados en \file{RAGAS} y \file{ARES}, verificando el correcto funcionamiento de la generación de preguntas, construcción de conjuntos de evaluación, cálculo de métricas y ejecución de los distintos \eng{pipelines} de evaluación. Se han incluido pruebas destinadas a validar la gestión de tareas asíncronas, el sistema de priorización de recursos utilizado para favorecer las consultas \eng{RAG} frente a procesos en segundo plano y los mecanismos de gestión de errores y excepciones.

La ejecución de las pruebas unitarias se puede realizar utilizando:

\begin{tcolorbox}[

	colframe=black!30,

\subsection{Pruebas de integración}

Las pruebas de integración verifican que los distintos componentes del sistema funcionen de manera correcta conjuntamente. Su principal diferencia con las pruebas unitarias es que no validan funciones aisladas sino la interacción entre los componentes. Estas pruebas permiten detectar errores derivados de la configuración del entorno, incompatibilidades entre servicios o problemas de comunicación entre componentes.

En el proyecto se han realizado pruebas destinadas a validar la comunicación entre la aplicación \file{Flask}, la base de datos relacional, la base de datos vectorial \file{Qdrant} y los modelos ejecutados mediante \file{Ollama}. También, se ha comprobado el correcto funcionamiento del flujo de indexación documental, recuperación semántica y generación de respuestas.

Cubren los distintos módulos funcionales de la aplicación mediante la ejecución de peticiones reales sobre los \file{Blueprints} de la aplicación. De esta forma se verifica el comportamiento conjunto de las rutas, formularios, servicios, modelos de datos y mecanismos de control de acceso implementados. Se han desarrollado pruebas específicas para validar los \eng{pipelines} de evaluación \eng{RAG} comprobando el correcto funcionamiento de lso distintos puntos de entrada, la generación de preguntas, la construcción de conjuntos de evaluación y la ejecución de los procesos de evaluación. También se han incluido pruebas para verificar el tratamiento de errores, el funcionamiento de los decoradores de autorización y la correcta integración entre los distintos componentes internos de la aplicación.

Para ejecutar estas pruebas se utiliza el contenedor definido para las pruebas dentro del entorno \file{Docker}. De esta manera las pruebas se ejecutan en un entorno equivalente al utilizado durante el despliegue de producción, garantizando que las interacciones entre servicios y dependencias externas se comporten de forma consistente.

\subsection{Pruebas de interfaz}

Además, de las pruebas automáticas del \eng{backend} se realizaron pruebas funcionales de la interfaz \eng{web} utilizando la herramienta \file{Katalon Recorder}. Estas pruebas permiten validar los principales flujos de interacción del usuario simulando acciones reales sobre la aplicación.

Para cada caso de prueba se verificó tanto la correcta navegación entre páginas como la aparición de los mensajes de confirmación y error correspondientes. La automatización mediante \file{Katalon Recorder} permite repetir las pruebas de forma sistemática tras la incorporación de nuevas funcionalidades o modificaciones en la interfaz. Los casos de prueba definidos fueron los siguientes:
\begin{enumerate}
\tightlist
	\item CU-01 Registro: Registro de un nuevo usuario verificando la creación correcta de la cuenta.
	\item CU-02 Inicio de sesión: Inicio de sesión con credenciales válidas y acceso a la página principal.
	\item CU-03 Recuperación contraseña: Recuperación de contraseña mediante correo electrónico.
	\item CU-04 Indexación documental: Carga de un documento y comprobación de su incorporación al sistema.
	\item CU-05 Consulta: Realización de una consulta RAG y verificación de la generación de una respuesta.
	\item CU-06 Historial: Consulta del historial y visualización de preguntas previamente realizadas.
	\item CU-07 Administración: Acceso al panel de administración y comprobación de las funcionalidades de gestión disponibles para usuarios administradores.
\end{enumerate}

\subsection{Ejecución automática de las pruebas}

Con el objetivo de simplificar la validación de sistema se ha desarrollado un \eng{script} (\file{run\_tests.sh}) encargado de ejecutar todas las pruebas de manera secuencial y generar informes de cobertura. 

Este proceso consiste en eliminar los archivos de cobertura de ejecuciones anteriores. Posteriormente se ejecutan las pruebas unitarias, de integración y de interfaz. Estas ejecuciones se hacen utilizando la herramienta \file{coverage}, lo que permite medir la parte del código que está cubierta por las pruebas. Cuando finalizan las pruebas el \eng{script} genera distintos informes:
\begin{itemize}
\tightlist
	\item \file{coverage.xml}: informe en formato \file{XML} utilizado por herramientas externas de análisis de calidad como \file{SonarCloud}.
	\item Informe por consola: muestra el porcentaje de cobertura alcanzado por cada módulo de la aplicación.
	\item Directorio \file{coverage/}: informe \file{HTML} navegable que permite visualizar línea por línea el código cubierto por las pruebas.
\end{itemize}

La ejecución completa de la batería de pruebas puede realizarse mediante el siguiente comando:
