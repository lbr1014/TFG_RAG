\section{Estructura de directorios}

El proyecto se organiza en tres bloques principales, la aplicación final (PythIA), los prototipos desarrollados durante la fase experimental y la documentación del Trabajo de Fin de Grado.

{\footnotesize
\dirtree{%
	.1 /.
	.2 .github/. 
	.3 workflows/.
	.4 build.yml $\rightarrow$ flujo de integración continua del proyecto, ejecuta las comprobaciones automáticas definidas para GitHub Actions, incluyendo el análisis de calidad con SonarCloud.
	.2 app/.
	.3 PythIA/ $\rightarrow$ versión final de la aplicación web desarrollada con Flask.
	.2 docs/.
	.3 LaTeX/ $\rightarrow$ documentación técnica y memoria del proyecto en formato LaTeX, junto con los documentos PDF generados.
	.2 prototipos/ $\rightarrow$ carpeta con archivos de prueba desarrollados durante el proyecto y versiones antiguas de la aplicación.
	.2 .gitignore $\rightarrow$ archivo que indica qué ficheros o carpetas deben ser ignorados por Git y no subirse al repositorio.
	.2 sonar-project.properties $\rightarrow$ archivo de configuración para el análisis de calidad del código mediante SonarCloud.
	.2 README.md $\rightarrow$ archivo principal de documentación del proyecto, explica qué es PythIA y cómo desplegar la aplicación en local.
}}

El directorio \file{app/PythIA} contiene la versión final de la aplicación. En ella se encuentra la aplicación Flask organizada siguiendo una separación entre controladores, modelos, servicios, recursos estáticos y plantillas \file{HTML}. También incluye la configuración necesaria para ejecutar la aplicación en contenedores Docker, usando Gunicorn como servidor de aplicación y Nginx como \eng{proxy} inverso.

Además, esta carpeta contiene los \eng{tests} unitarios y de integración, que permiten comprobar el correcto funcionamiento de las rutas, modelos, servicios y funcionalidades principales. Las migraciones permiten mantener actualizada la estructura de la base de datos en los distintos entornos.

El directorio \file{controllers} contiene las rutas de Flask, organizadas mediante \eng{Blueprints}. Esta organización permite dividir la aplicación en módulos funcionales independientes, manteniendo alta cohesión dentro de cada módulo y reduciendo el acoplamiento entre funcionalidades.

{\footnotesize
\dirtree{%
	.1 controllers/.
	.2 admin/ .
	.3 routes.py $\rightarrow$ registro, inicio de sesión, cierre de sesión y recuperación de contraseña.
	.2 auth/ .
	.3 routes.py $\rightarrow$ gestión de usuarios y documentos.
	.2 main/ .
	.3 routes.py $\rightarrow$ rutas generales de navegación, la página principal y el historial.
	.2 rag/ .
	.3 routes.py $\rightarrow$ interfaz de consulta al RAG y la gestión de respuestas.
}}

El directorio \file{model} contiene las entidades persistentes de la aplicación. Estas clases representan los datos almacenados en PostgreSQL. Permiten gestionar usuarios, documentos, conslutas, \eng{chunks}, \eng{embeddings} y estados de procesos asíncronos.

{\footnotesize
\dirtree{%
	.1 model/.
	.2 user.py $\rightarrow$ define la entidad de usuarios del sistema (autenticación, roles y gestión de cuentas).
	.2 documento.py $\rightarrow$ representa los documentos de licitaciones almacenados, incluyendo metadatos y estado de procesamiento.
	.2 consulta.py $\rightarrow$ modela las consultas realizadas por los usuarios al sistema RAG y sus resultados.
	.2 chunk.py $\rightarrow$ define los fragmentos de texto generados a partir de los documentos para su procesamiento.
	.2 embedding.py $\rightarrow$ representa los vectores generados a partir de los chunks para su almacenamiento en la base de datos vectorial.
	.2 consulta\_chunk.py $\rightarrow$ modela la relación entre consultas y los chunks utilizados para generar la respuesta.
	.2 job\_state.py $\rightarrow$ clase base para gestionar el estado de tareas largas o procesos en segundo plano.
	.2 rag\_query\_state.py $\rightarrow$ controla el estado de ejecución de las consultas al sistema RAG.
	.2 web\_scraping\_state.py $\rightarrow$ gestiona el estado de los procesos de web scraping.
	.2 vector\_update\_state.py $\rightarrow$ controla el estado de actualización de la base de datos vectorial.
	.2 markdown\_conversion\_state.py $\rightarrow$ gestiona el estado de los procesos de conversión de documentos a formato Markdown.
}}

El directorio \file{services} agrupa la lógica de negocio de la aplicación. A diferencia de los controladores, que reciben las peticiones HTTP, los servicios ejecutan las operaciones internas.

	.2 async\_tasks.py $\rightarrow$ ejecución de procesos largos en segundo plano.
	.2 documentos.py $\rightarrow$ gestión de documentos cargados en el sistema.
	.2 markdown/.
	.3 Conversion\_markdown.py $\rightarrow$ implementa la conversión de documentos PDF a formato Markdown, incluyendo limpieza y estructuración del texto.
	.2 rag/.
	.3 PrototipoRAGpy $\rightarrow$ implementación del pipeline RAG (recuperación de chunks relevantes y generación de respuestas con el LLM).
	.3 default\_prompts.py $\rightarrow$ define los prompts por defecto utilizados para interactuar con los modelos de lenguaje.
	.3 service.py $\rightarrow$ orquesta el funcionamiento del sistema RAG integrando recuperación, generación y gestión de respuestas.
	.2 web\_scraping/.
	.3 DescargarPliegos.py $\rightarrow$ script encargado de descargar los documentos PDF de licitaciones.
	.3 PliegosPlaywrightAsincrono.py $\rightarrow$ implementa el web scraping asíncrono mediante Playwright para obtener datos de licitaciones.
	.3 pliegos\_pdfs.json $\rightarrow$ almacena los enlaces y metadatos de los documentos PDF obtenidos.
	.3 resultados\_playwright\_asincrono\_servidor.json $\rightarrow$ contiene los resultados del scraping (enlaces, información extraída, etc.).
	.2 markdown\_conversion\_state.py $\rightarrow$ control de estado de las tareas de conversión a Markdown.
	.2 vector\_update\_state.py $\rightarrow$ control de estado de las tareas de actualización de la base de datos vectorial.
	.2 web\_scraping\_state.py $\rightarrow$ control de estado de las tareas del web scraping.
}}

En el directorio \file{resources} se encuentran los elementos de la capa de presentación de la aplicación. Incluye tanto las plantillas HTML renderizadas dinámicamente como los recursos estáticos utilizados por la interfaz web.

Dentro de él se encuentra el directorio \file{tempaltes} y el \file{static}. En el directorio \file{tempaltes} se encuentran las plantillas HTML renderizadas por Flask mediante el motor de plantillas Jinja2, las cuales forman la capa de vista de la aplicación. En el directorio \file{static} se encuentran los recursos estáticos que utiliza la aplicación web. Estos archivos no son procesados por el servidor y se envían directamente al navegador.

Esta separación de la aplicación permite tener tres módulos separados claramente para facilitar el mantenimiento de la interfaz y permitir cambios en el diseño sin afectar a la lógica del sistema.. El primero contiene la lógica de negocio (\eng{backend}) definida en \file{controllers} y \file{services}, el segundo los datos (modelo) definida en \file{model} y el tercero la interfaz (vista) definida en \file{templates} y \file{static}.

La carpeta prototipos recoge las pruebas y versiones iniciales desarrolladas antes de llegar a la aplicación final. En ella se encuentran los primeros experimentos con RAG, la creación de la base de datos vectorial, la consulta mediante terminal, la descarga automática de pliegos, la conversión de documentos PDF a Markdown y las primeras versiones de la aplicación web.

	.2 BaseDatos.py $\rightarrow$ script para crear la base de datos vectorial en Qdrant a partir de los documentos PDF.
	.2 PrototipoRAG.py $\rightarrow$ primer prototipo completo del sistema RAG, con carga de documentos, generación de embeddings, búsqueda semántica y generación de respuestas con Ollama.
	.2 Prompt.py $\rightarrow$ interfaz por terminal para realizar preguntas al prototipo RAG.
	.2 PruebaBaseDatos.py $\rightarrow$ script para comprobar qué documentos y fragmentos han sido almacenados en la base de datos vectorial.
	.2 Flask/ $\rightarrow$ primer prototipo de la aplicación web Flask, con rutas, login, administración y uso de SQLite.
	.2 Flask\_docker/ $\rightarrow$ versión posterior del prototipo Flask preparada para ejecutarse con Docker, Gunicorn, Nginx, PostgreSQL y Qdrant.
	.2 Markdown/ $\rightarrow$ pruebas de conversión de documentos PDF a Markdown mediante distintas técnicas, incluyendo OCR y modelos locales.
	.2 Web Scraping/ $\rightarrow$ scripts para recopilar licitaciones y descargar pliegos desde la Plataforma de Contratación del Sector Público.
	.2 tokenizers/ $\rightarrow$ pruebas de tokenización, división de documentos largos y generación de resúmenes.
	.2 pliegos/ $\rightarrow$ documentos PDF utilizados como datos de prueba.
	.2 qdrant\_data/ $\rightarrow$ datos generados por la base de datos vectorial Qdrant.
	.2 requirements.txt $\rightarrow$ dependencias generales de los prototipos.
	.2 pyproject.toml $\rightarrow$ configuración del proyecto para Poetry.
	.2 poetry.lock $\rightarrow$ bloqueo de versiones de dependencias.
	.2 poetry.toml $\rightarrow$ configuración local de Poetry.
	.2 .gitignore $\rightarrow$ archivos ignorados dentro de la carpeta de prototipos.
