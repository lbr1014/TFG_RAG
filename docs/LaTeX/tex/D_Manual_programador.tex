\section{Manual del programador}
Este apartado tiene como objetivo proporcionar a futuros desarrolladores que necesiten desplegar, modificar o ampliar la aplicación.

El proyecto implementa una aplicación basada en una arquitectura RAG (\eng{Retrieval Augmented Generation}. Orientada a la gestión y consulta inteligente de documentación, utilizando tecnologías de procesamiento de lenguaje natural, recuperación semántica y generación de respuestas mediante modelos de inteligencia artificial.

El sistema ha sido desarrollado principalmente en Python utilizando el framework Flask para la capa web y diferentes librerías relacionadas con embeddings, recuperación documental y modelos de lenguaje. Además, se emplea Docker para facilitar la ejecución del entorno y garantizar la reproducibilidad del sistema.

%%%%%%%%%%%%%%%
El desarrollo del proyecto se ha realizado utilizando las siguientes herramientas y tecnologías:
\begin{itemize}
	\item Lenguaje de programación: Python 3.
	\item Framework \eng{backend}: Flask.
	\item Base de datos relacional: SQLite/MySQL según entorno.
	\item Sistema de migraciones: Flask-Migrate.
	\item Sistema de plantillas: Jinja2.
	\item Gestión de dependencias: pip.
	\item Contenedores: Docker y Docker Compose.
	\item Control de versiones: Git y GitHub.
	\item Entorno de desarrollo: Visual Studio Code.
	\item Librerías de IA y recuperación semántica: LangChain, modelos de embeddings y Qdrant.
\end{itemize}

La estructura general del proyecto se encuentra organizada siguiendo una arquitectura modular que separa la lógica de negocio, las rutas, los formularios y las funcionalidades relacionadas con el sistema RAG. Concretamente la aplicación sigue una arquitectura cliente-servidor multicapa. La capa de presentación está formada por las plantillas HTML renderizadas mediante Flask y Jinja2, mientras que la lógica de negocio se implementa en los controladores y servicios internos. El componente principal del proyecto es el sistema RAG. Este sistema combina recuperación semántica de información y generación de respuestas utilizando modelos de lenguaje.

El núcleo del sistema se basa en un flujo RAG compuesto por las siguientes etapas:
\begin{enumerate}
	\item Carga y procesamiento de documentos.
	\item Conversión de documentos a formato Markdown.
	\item Fragmentación del contenido textual.
	\item Generación de \eng{embeddings} semánticos.
	\item Indexación de fragmentos en la base vectorial.
	\item Recuperación de contexto relevante ante una consulta.
	\item Generación de respuesta mediante un modelo de lenguaje.
\end{enumerate}

Por otra parte, la aplicación implementa un sistema de Blueprints de Flask para dividir las distintas áreas funcionales del sistema.

\begin{enumerate}
	 \item Módulo de autenticación (\file{auth}): es el encargado de registrar usuarios, el inicio y cierre de sesión, la recuperación de contraseña, la gestión de sesiones y la validación de credenciales.
	 \item Módulo principal (\file{main}): se encarga de la página principal, así como de la navegación general. Gestiona el historial de las consultas y permite la visualización de la información del usuario.
	 \item Módulo RAG (\file{rag}): contiene la lógica para subir documentos, convertirlos a Markdown, procesar semánticamente, recuperar el contexto para la consulta del usuario y generar las respuestas llamando a un LLM.
	 \item Módulo de administración (\file{admin}): permite gestionar a los usuarios, asignarles permisos y administrar el sistema (carga de documentos, actualización de la base de datos vectorial, etc).
\end{enumerate}

El flujo de datos del procesamiento documental tras la carga de un documento por parte de un usuario en el sistema es:

\begin{itemize}
	\item Validación del archivo.	
	\item Conversión del documento a texto Markdown.
	\item Limpieza y normalización del contenido.
	\item División en fragmentos de tamaño controlado.
	\item Generación de \eng{embeddings} para cada fragmento.
	\item Almacenamiento de \eng{embeddings} en la base vectorial.
\end{itemize}

El sistema mantiene la trazabilidad entre los fragmentos indexados y el documento original.

El flujo de datos de la recuperación semántica ante una consulta del usuario (ver imagen ~\ref{fig:Diagramas/diagrama_flujo} ):

\begin{itemize}
	\item Se genera un \eng{embedding} de la pregunta.
	\item Se realiza una búsqueda vectorial.
	\item Se recuperan los fragmentos más relevantes.
	\item El contexto recuperado se envía al modelo de lenguaje.
	\item El modelo genera una respuesta contextualizada.
\end{itemize}

Este enfoque permite reducir alucinaciones y mejorar la precisión de las respuestas generadas.

El sistema utiliza una base de datos relacional para almacenar:

\begin{itemize}
	\item Usuarios.
	\item Roles.
	\item Historial de consultas.
	\item Metadatos de documentos.
	\item Información de indexación.
\end{itemize}

Las migraciones de la base de datos se gestionan mediante Flask-Migrate y Alembic.

Para crear una nueva migración se utiliza:

\texttt{flask db migrate -m "descripcion"}

Y para aplicarla:

\texttt{flask db upgrade}

La persistencia vectorial se realiza mediante Qdrant, donde se almacenan los \eng{embeddings} generados durante la indexación.

El proyecto utiliza un amplio conjunto de dependencias organizadas por módulos funcionales. Estas librerías permiten implementar la aplicación web, la lógica del sistema RAG, la gestión documental, el procesamiento semántico y el despliegue completo del entorno.

Las dependencias del proyecto se encuentran definidas principalmente en el archivo \file{requirements.txt}.Para instalar todas las dependencias necesarias:

\texttt{pip install -r requirements.txt}

El núcleo de la aplicación se ha desarrollado utilizando Flask y diferentes extensiones asociadas.

\begin{itemize}
	\item Flask: \eng{framework} k principal utilizado para construir la aplicación web y gestionar rutas, peticiones HTTP y renderizado de plantillas.
	\item Jinja2: motor de plantillas utilizado por Flask para generar páginas HTML dinámicas.
	\item Werkzeug: conjunto de utilidades internas de Flask relacionadas con seguridad, subida de archivos, gestión de peticiones y servidor WSGI.
	\item itsdangerous: librería utilizada para la generación de \eng{tokens} seguros empleados en validaciones y recuperación de contraseñas.
	\item Flask-Login: extensión encargada de gestionar sesiones de usuario, autenticación y control de usuarios autenticados.
	\item Flask-WTF: integración de WTForms con Flask para facilitar la gestión de formularios.
	\item WTForms: librería para definición y validación de formularios web.
	\item Flask-Mail: extensión utilizada para el envío de correos electrónicos desde la aplicación.
	\item email-validator: validación de direcciones de correo electrónico introducidas por el usuario.
	\item python-dotenv: carga automática de variables de entorno definidas en archivos \file{.env}.
	\item gunicorn: servidor WSGI utilizado para ejecutar Flask en entornos de producción.
\end{itemize}

\begin{itemize}

	\item pydantic-core: motor interno optimizado utilizado por Pydantic.
	\item typing\_extensions y typing-inspection: librerías auxiliares para anotaciones avanzadas de tipos.
	\item coverage: herramienta utilizada para medir la cobertura de los tests.
	\item unittest: framework de pruebas unitarias incluido en la librería estándar de Python.
\end{itemize}

	\item Nginx: proxy inverso encargado de gestionar conexiones HTTP y redirigir peticiones hacia Gunicorn.
	\item Gunicorn: servidor WSGI utilizado para ejecutar Flask en producción.
	\item PostgreSQL Docker image: contenedor de la base de datos relacional.
	\item Qdrant Docker image: contenedor de la base de datos vectorial.
	\item Ollama Docker image: contenedor encargado de ejecutar modelos LLM locales.
	\item Playwright Python Docker image: imagen base utilizada para las tareas de \eng{webs craping} automatizado.
\end{itemize}

La configuración del sistema se realiza mediante variables de entorno, permitiendo desacoplar los parámetros sensibles y específicos del entorno de ejecución respecto al código fuente de la aplicación. Esto permite separar la configuración y lógica de negocio, adaptar fácilmente el sistema a distintos entornos, evitar exponer credenciales privadas en el repositorio, simplificar el despliegue medianet \file{Docker Compose} y mantener configuraciones reutilizables entre desarrollo y producción. \section{Compilación, instalación y ejecución del proyecto}

En este apartado se describen los pasos necesarios para compilar, instalar y ejecutar el proyecto PythIA. El proyecto ha sido diseñado para ejecutarse principalmente mediante contenedores Docker, lo que permite reproducir el entorno completo de forma homogénea y transparente al usuario, independientemente del sistema operativo utilizado. La aplicación integra múltiples servicios que gracias a la utilización de Docker Compose se pueden desplegar de manera coordinada. Por ejemplo, la aplicación Flask, la base de datos PostgreSQL, la base de datos vectorial Qdrant, los modelos LLM ejecutado mediante Ollama y el servidor Nginx utilizado como proxy inverso.
